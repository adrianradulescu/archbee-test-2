Is a collaborative documentation platform. Archbee sells software to other companies as a subscription service, helping companies turn documentation into clear, accessible knowledge and instant answers instead of PDFs or messy Google Docs. Archbee is a complete product documentation platform with advanced features such as reusable content, versioning, document revision history, and collaboration features to keep product and engineering teams in sync and the documentation up-to-date. # API

API’S

**API**: An API, or application programming interface, is a set of rules and protocols for building and interacting with software applications. Is basically a way for two apps/systems to talk to each other. In general, an API (or Application Programming Interface) provides an interface between two systems. It’s like a cog that allows two systems to interact with each other. In this case, the two systems are computers that interact programmatically through the API.

By defining a clear set of methods and tools, APIs ensure that different software can reliably interact regardless of their underlying architecture or technology. Because of this, APIs have played a critical role in the growth of the internet, enabling cloud services, mobile application development, and the integration of web-based services. Today, APIs are much more than simple code libraries. They are sophisticated tools that define the way we interact with technology.

- You (app) send a request
- The API delivers it to another system
- That system sends a response back

When you see a YouTube video embedded on a site, the site itself doesn’t host the video. Instead, the embed code contains API requests that get the video from YouTube’s hosting centers. When you embed a Twitter widget on your site, the widget’s APIs pull in tweets from a Twitter server onto your site

The next time you push a button in some interface, think about what’s going on underneath. Most likely, the button is internally wired to make calls to an external service for the information needed. That call is made through an API that performs a request and receives a response.

Despite the variety of APIs, the defining characteristic of nearly all developer documentation is that it involves documenting some type of API. This is why “API documentation” and “developer documentation” are used somewhat synonymously. APIs make life easier for developers (who are consuming the API) because the APIs perform functions or other tasks in more efficient ways.

One way to sort the different types of APIs is to categorize them into two general buckets: web service APIs versus native library APIs.

**Web service APIs:**  send and receive messages across the web using HTTP to transport the request and response; web service APIs are language agnostic. Web services include various types of APIs, including both REST and SOAP APIs.
**Native library APIs:** involve incorporating code directly into your project for the desired functionality; native library APIs are language-specific.Refer to code libraries (for example, JAR files) that developers add directly to their projects to provide additional functionality through classes or other functions that can be called locally. These APIs are specific to a programming language — e.g., Java, C++, Python, Ruby, .NET, and so on. With native library APIs, the functions are incorporated locally within the code to expand the operations you can perform within your project, usually without requiring you to access resources in the cloud. Native library APIs require you to be familiar with the programming language and tend to be the most challenging type of API to document for technical writers.

**SOAP** (Simple Object Access Protocol) APIs are web services that rely on a strict XML protocol to define the message exchange format for requests and responses. SOAP is common with financial APIs and regulated industries, though it has largely been replaced by REST in popularity. As a standardized protocol, SOAP’s XML message format is usually defined through a WSDL (Web Services Description Language) file that specifies the allowed elements and attributes in the message exchange. The WSDL file is machine-readable and used by the servers interacting with each other to facilitate the communication.

**RPC-based APIs:** RPC stands for Remote Procedure Call. RPC-based APIs are web services that call a method on a remote server by delivering an encoded message through HTTP. The encoded message format might be XML for [XML-RPC APIs](https://en.wikipedia.org/wiki/XML-RPC) or JSON for [JSON-RPC APIs](https://en.wikipedia.org/wiki/JSON-RPC), but in both cases, the message travels to the remote server via HTTP like other web services. The methods on the remote servers can be in any language. For example, an XML-RPC API can call a Java or Python or C++ method.

**gRPC APIs:** gRPC APIs are web services similar to RPC-based APIs in that the web service calls a function or runs a procedure on a remote server; however, gRPC uses [protocol buffers](https://developers.google.com/protocol-buffers/docs/overview#what-are-protocol-buffers) (specified in .proto files) rather than XML or JSON as the message exchange format. The protocol buffer lets you define the structure for your data and the way to convert (serialize) the data to be consumed by the receiving server. Protocol buffers are lighter and more efficient than XML. gRPC APIs were developed by Google and published as an open-source platform. See [gRPC APIs](https://grpc.io/docs/guides/) for details.

**REST APIs**: REST (Representational State Transfer) are web services that let you make requests for resources through URL paths. You also supply the operation to be performed with the path (e.g., GET, CREATE, DELETE). As with other web service APIs, the requests and responses travel via HTTP across the web, and the servers receiving the requests are language agnostic about the request (not required to be a specific programming language). Responses are typically returned in JSON or XML formats. REST APIs have many different paths (endpoints) with various parameters you can configure to determine the results you want. This course mainly focuses on REST APIs. See [What is a REST API?](https://idratherbewriting.com/learnapidoc/docapis_what_is_a_rest_api.html) for details.

**GraphQL APIs**: GraphQL APIs are web services developed by Facebook that let users dynamically query for results they need through a single path (endpoint). GraphQL eliminates the need for multiple request URLs or other post-filtering on the returned results to get what you need. Your query retrieves only the needed data, allowing the request and response to be fast and specific. See [graphql.org](https://graphql.org/) for more detail. Also see [If I am learning to write developer documentation, should GraphQL be on my radar?](https://idratherbewriting.com/blog/graphql-relevance-and-documentation-strategies/)

**Voice Assistant APIs**: Voice Assistant APIs are used with voice assistants such as Alexa. These APIs originate from the cloud and call an endpoint based on natural language processing of voice commands spoken by users. This is a case where APIs operate behind the scenes in the cloud, and developers create code, such as in a Lambda function (cloud computing), that handles incoming requests sent from the voice assistant API.

**Internet of Things (IoT) APIs**: IoT APIs are used by physical devices (such as sensors or wearables) that transmit or receive data to connect the device to an online network. For example, a thermostat sensor in a room might transmit the temperature to a central controller (such as with Nest) via an IoT API.

**REST API’S**

REST (Representational State Transfer) uses HTTP as the transport protocol for the message requests and responses. However, unlike SOAP, REST is an architectural style, not a standard protocol. This is why REST APIs are sometimes called RESTful APIs — REST is a general style that the API follows. REST APIs follow the same protocol as the web. When you open a browser and type a website URL (such as [https://idratherbewriting.com](https://idratherbewriting.com)), you’re actually making a GET request for a resource on a server. The server responds with the content and the browser makes the content visible.

REST APIs focus on resources ( things, rather than actions) and ways to access the resources. Resources are typically different types of information. You access the resources through URLs (Uniform Resource Locators), just like going to a URL in your browser retrieves an information resource. The URLs are accompanied by a method that specifies how you want to interact with the resource.

Common methods include GET (read), POST (create), PUT (update), and DELETE (remove). The endpoint usually includes query parameters that specify more details about the representation of the resource you want to see. For example, you might specify (in a query parameter) that you want to limit the display to 5 instances of the resource.
Here’s what a sample endpoint might look like:

```javascript
http://apiserver.com/homes?limit=5&format=json
```

The endpoint shows the whole path to the resource. However, in documentation, you usually separate this URL into more specific parts:

**The base path** (or base URL or host) refers to the common path for the API. In the example above, the base path is [http://apiserver.com](http://apiserver.com).
**The endpoint** refers to the end path of the endpoint. In the example above, /homes.
The **?limit=5\&format=json** part of the endpoint contains query string parameters for the endpoint.

The terminology of “GET requests” and “message responses” transported over “HTTP protocol” might seem unfamiliar, but this is just the official REST terminology to describe what’s happening. Because you’ve used the web, you’re already familiar with how REST APIs work — the web itself essentially follows a RESTful style.

If you open a browser and go to [https://idratherbewriting.com](https://idratherbewriting.com/), you’re really using HTTP protocol (https\://) to submit a GET request to the resource available on a web server. The response from the server sends the content at this resource back to you using HTTP. Your browser is just a client that makes the message response look pretty.

An important aspect of REST APIs, especially in the context of documentation, is that they don’t use a WSDL file to describe the elements and parameters allowed in the requests and responses.
To understand how to interact with a REST API, you have to read the documentation for the API. The need to read the docs makes the technical writer’s role extremely important with REST APIs.
Some formal specifications — for example, [OpenAPI](https://idratherbewriting.com/learnapidoc/pubapis_openapi_intro.html) and [RAML](https://idratherbewriting.com/learnapidoc/pubapis_raml.html) — have been developed to describe REST APIs. When you describe your API using the OpenAPI or RAML specification, tools that can read those specifications (such as [Swagger UI](https://idratherbewriting.com/learnapidoc/pubapis_swagger.html) or the [RAML API Console](https://idratherbewriting.com/learnapidoc/pubapis_raml.html#apiconsole)) will generate an interactive documentation output.

Documentation for REST APIs is important because REST follows an architectural style rather than an exact protocol standard. Each REST API functions a bit differently. There isn’t a single way of doing things, and this flexibility and variety fuel the need for accurate and clear documentation.

**POSTMAN**
When you’re testing endpoints with different parameters, you can use one of the many GUI REST clients available to make the requests. (By “GUI,” I mean there’s a graphical user interface with boxes and buttons for you to click.) You can also use curl (which we’ll cover soon), but GUI clients tend to simplify testing with REST APIs.

With a GUI REST client, you can:

- Save your requests (and numerous variations) in a way that’s easy to run again
- More easily enter information in the right format
- See the response in a prettified JSON view or a raw format
- Easily include header information

With a GUI REST client, you won’t have to worry about getting curl syntax right and analyzing requests and responses from the command line.

Some popular GUI clients include the following:

- Postman
- Insomnia
- Paw
- Advanced REST Client (Chrome browser extension)

Of the various GUI clients available, **Postman** is probably the best option, since it allows you to save both calls and responses, is free, works on both Mac and PC, and is easy to configure.

**CURL**

One of the advantages of REST APIs is that you can use almost any programming language to call the endpoint. The endpoint is simply a resource located on a web server at a specific path.Each programming language has a different way of making web calls. Rather than exhausting your energies trying to show how to make web calls in Java, Python, C++, JavaScript, Ruby, and so on, you can just show the call using curl.

Curl provides a generic, language-agnostic way to demonstrate HTTP requests and responses. Users can see the format of the request, including any headers and other parameters. Your users can translate this into the specific format for the language they’re using.

**JSON FORMAT**

*JSON is the most common format for responses from REST APIs.*

JSON stands for JavaScript Object Notation. It’s the most common way REST APIs return information. Although some APIs return information in both JSON and XML, if you’re trying to parse through the response and render it on a web page, JSON fits much better into the existing JavaScript + HTML + CSS technology that powers most web pages. With JavaScript, you can easily parse through the JSON and integrate it into your web content.

JSON has two types of basic structures:

An **object** is a collection of key-value pairs, surrounded by curly braces:



The key-value pairs are each put into double quotation marks when both are strings. If the value is an integer (a whole number) or Boolean (true or false value), omit the quotation marks around the value. Each key-value pair is separated from the next by a comma.

An **array** is a list of items, surrounded by brackets:



The list of items can contain strings, numbers, booleans, arrays, or other objects. With integers or booleans, you don’t use quotation marks.

Just remember, objects are enclosed by curly braces \{ } and contain key-value pairs. Sometimes those values are arrays. Arrays are lists and are enclosed by square brackets \[ ]. It’s common for arrays to contain lists of objects, and for objects to contain arrays.

# PART 1 — Understanding Your Role

## 1. What Customer Support in a SaaS Company Actually Is

Customer support in a SaaS (Software as a Service) company is much more than simply answering tickets. A support team acts as the bridge between the customer and the company. Support agents help users solve problems, understand the product, and continue using the platform successfully.

In a company like Archbee, support is especially important because many customers are technical users such as developers, technical writers, product teams, and engineering teams. These users often work with APIs, domains, GitHub integrations, authentication systems, and documentation workflows. Because of this, support agents need a combination of:

- communication skills
- troubleshooting skills
- technical curiosity
- product knowledge

A strong support agent is not necessarily the person who knows every technical answer immediately. Instead, a strong support agent:

- knows how to investigate
- asks good questions
- understands how the product works
- communicates clearly
- stays organized
- learns patterns over time

One of the most important mindset shifts is understanding that support is not about “guessing.” It is about investigation.

When customers report issues, your role is often to:

1. **understand the problem**
2. **gather useful information**
3. **reproduce the issue if possible**
4. **identify whether it is:**
   - **user confusion**
   - **configuration issue**
   - **limitation**
   - **bug**
5. **provide guidance or escalate correctly**

This means support work becomes easier as your exposure to tickets increases.

At first, technical terms like:

- JWT
- API
- DNS
- GitHub sync
- custom domains
- CSS may feel intimidating.

However, over time you begin seeing the same concepts repeatedly. Eventually, you stop seeing them as “advanced topics” and instead see them as recurring systems with predictable behaviors.

Another important thing to understand is the relationship between departments.

### Support vs Engineering

Engineering builds and fixes the product. Support communicates with customers and investigates issues.

Support should avoid escalating vague tickets like:

- “It doesn’t work.”

Instead, **strong escalations include:**

- **steps to reproduce**
- **screenshots**
- **browser information**
- **affected URLs**
- **expected behavior**
- **actual behavior**

The more context you provide, the easier it is for engineering to investigate.

### Support vs Customer Success

Customer success focuses more on customer relationships, onboarding, retention, and long-term satisfaction.

Support focuses more on:

- troubleshooting
- issue resolution
- technical guidance
- product behavior

In many SaaS companies, these areas overlap.

### Support vs Product Team

The product team decides what features are built. Support often becomes the voice of the customer by identifying:

- recurring frustrations
- feature requests
- UX confusion
- workflow pain points

This is why ticket patterns matter.

Over time, you will start noticing recurring categories such as:

- custom domain problems
- publishing issues
- permissions confusion
- GitHub sync issues
- formatting issues
- search behavior
- API misunderstandings

Recognizing these patterns is one of the fastest ways to grow technically.

## 2. Understanding Archbee as a Product

To become strong in support, you must deeply understand what Archbee actually is.

Archbee is a documentation platform that allows teams to create, organize, publish, and manage documentation.

Customers may use Archbee for:

- public documentation
- API documentation
- internal company documentation
- product manuals
- onboarding guides
- developer portals
- knowledge bases
- white-label documentation sites

This means the platform serves both technical and non-technical users.

### Common Customer Types

### Technical Writers

Technical writers focus on creating organized, understandable documentation.

They care about:

- structure
- readability
- formatting
- navigation
- publishing
- search
- consistency

### Developers

Developers often care about:

- APIs
- GitHub integrations
- authentication
- embeds
- custom domains
- code blocks
- developer documentation
- automation

Developers may use more technical terminology and expect precise explanations.

### Product Teams

Product teams use documentation to help customers understand products and workflows.

They often care about:

- user experience
- organization
- collaboration
- branding
- publishing workflows

### Understanding Documentation Platforms

A documentation platform is essentially a system for:

- creating content
- storing content
- organizing content
- displaying content
- publishing content

Many customer issues happen because users misunderstand how these systems behave.

For example:

- draft vs published content
- caching delays
- permissions
- broken links
- search indexing
- domain propagation

Understanding these systems conceptually is more important than memorizing random fixes.

One of the best ways to improve quickly is to actively explore the platform yourself.

You should regularly:

- create test pages
- experiment with formatting
- test links
- publish/unpublish content
- test permissions
- explore settings
- reproduce customer reports

Hands-on learning is one of the fastest ways to build intuition.

## 3. Core Support Skills

Technical knowledge alone is not enough to become successful in support.

Many strong support agents succeed because they combine technical investigation with excellent communication.

### Clear Communication

Customers are often frustrated because:

- something is broken
- they are blocked
- they are confused
- deadlines are affected

Your communication should help reduce confusion.

Good support communication is:

- clear
- direct
- organized
- calm
- professional

Avoid:

- overly complicated explanations
- vague responses
- assumptions
- unnecessary jargon

Instead of:

- “There may be some backend inconsistencies.”

Prefer:

- “The issue may be related to caching or synchronization delays.”

### Asking Good Questions

One of the most valuable support skills is learning how to gather useful information.

Examples of useful questions:

- Can you share screenshots?
- What browser are you using?
- Can you reproduce the issue consistently?
- When did this begin?
- Does this happen for all users or only specific users?
- Are there any console errors?
- Can you share the affected URL?

Good questions speed up investigations dramatically.

### Troubleshooting Mindset

Troubleshooting is a process of elimination.

Instead of panicking, strong support agents think systematically:

- What changed?
- Is this reproducible?
- Is it permissions-related?
- Is it browser-related?
- Is it caching?
- Is it expected behavior?
- Is this a bug?

### Internal Notes

Internal notes are extremely valuable because they:

- help teammates understand investigations
- provide escalation context
- preserve troubleshooting history
- improve collaboration

Good internal notes are:

- concise
- organized
- factual
- actionable

### Escalations

Escalations should save engineering time.

A weak escalation:

- “Customer says domains are broken.”

A strong escalation:

- “Customer reports SSL issues on custom domain after updating CNAME records. Reproduced issue on Chrome. Domain resolves intermittently. Attached screenshots and affected URLs.”

### Continuous Learning

One of the most important things to understand is that technical confidence develops gradually.

You do not need to master everything immediately.

Most support growth comes from:

- exposure
- repetition
- curiosity
- investigating real tickets
- asking questions
- documenting learnings

Over time, the concepts that once felt intimidating begin feeling familiar.

That is how technical intuition is built.
